Spring中Bean的加载

发表于 2019-03-23 | 更新于 2019-03-24   |   2k 字 | 分类于 Spring 5   |   作者 Jakcie.Yang | 阅读次数：
bean 设计模式 Spring

概述

Spring通过资源加载器加载相应的XML文件，使用读取器读取资源加载器中的文件到读取器中，在读取过程中，解析相应的xml文件元素，转化为spring定义的数据结BeanDefinition，把相应的BeanDefinition注册到注册表中的RootBeanDefinition。注册表中包含的BeanDefinition的数据结构。

技术点

在Spring中，@Service、@Controller、@Repository、@Component默认都是单例的,可以配合@ComponentScan来加载bean。用了私有全局变量，若不想影响下次请求，就需要用到原型模式，即@Scope(“prototype”)

单例模式

单例模式就是Spring的IOC机制只创建该类的一个实例，每次请求，都会用这同一个实例进行处理，因此若存在全局变量，本次请求的值肯定会影响下一次请求时该变量的值。单例模式可能会有线程安全的问题。

原型模式

原型模式指的是每次调用时，会重新创建该类的一个实例，比较类似于我们自己自己new的对象实例。原型模式不存在线程安全的问题。

有状态会话bean

有公用属性，可能被不同的线程改变，每个用户有自己特有的一个实例，在用户的生存期内，bean保持了用户的信息，即“有状态”；一旦用户灭亡（调用结束或实例结束），bean的生命期也告结束。即每个用户最初都会得到一个初始的bean。

无状态会话bean

没有类的公用属性，不会同时被多个线程改变，bean一旦实例化就被加进会话池中，各个用户都可以共用。即使用户已经消亡，bean的生命期也不一定结束，它可能依然存在于会话池中，供其他用户调用。由于没有特定的用户，那么也就不能保持某一用户的状态，所以叫无状态bean。但无状态会话bean并非没有状态，如果它有自己的属性（变量），那么这些变量就会受到所有调用它的用户的影响，这是在实际应用中必须注意的。

加载过程图（Spring 5.1.5）

主要类
AbstractBeanFactory: 实现了依赖关系处理
TypeConverter: 实现了类型转换
AbstractAutowireCapableBeanFactory: 实现了bean的create过程
DefaultListableBeanFactory: 实现了BeanDefinition的管理

主要接口
FactoryBean：通过getObject生成bean实例
InitializingBean：在实例化后执行afterPropertiesSet方法
BeanClassLoaderAware：注入classLoad
BeanFactoryAware:注入一个BeanFactory
DisposableBean：销毁bean默认调用destroy方法

源码

  • AbstractBeanFactory中的获取bean方法getBean

  • getBean实际调用了核心方法doGetBean

  • getSingleton从Spring容器中获取单例Bean

  • doGetBean调用的主要方法getObjectForBeanInstance

  • getObjectForBeanInstance 调用的主要方法getObjectFromFactoryBean

  • getObjectFromFactoryBean调用的主要方法doGetObjectFromFactoryBean

  • doGetObjectFromFactoryBean调用factory的getObject返回bean对象实例
